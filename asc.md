## Ascension Phase

## 生成手順（reference for character data generation）

以下は、`internal/characters/<char>/data_gen.textproto`（例: `internal/characters/albedo/data_gen.textproto`）がどのように作られ、レベル別の値（Ascension table）が算出されるかの手順です。将来のキャラクター実装のリファレンスとしてAI/人が再現できるようにまとめています。

### 前提ファイルと役割
- `data_gen.textproto`: キャラクターの基礎値（`base_hp` / `base_atk` / `base_def`）、使用する成長曲線、各ascension段階の `promo_data`（ascensionで追加される差分値）を含む。これが最終的に `*_gen.go` に埋め込まれる。
- `*_gen.go`（例: `albedo_gen.go`）: `//go:embed data_gen.textproto` で `data_gen.textproto` を埋め込み、`prototext.Unmarshal` で `model.AvatarData` に変換して実行時に利用する自動生成ファイル。
- `pkg/model/curves.go`: 各レベルに対応する成長係数マップ（`AvatarGrowCurveByLvl`）。基礎値にこの係数を掛けて任意レベル時点の値を得る。

### 値の生成ロジック（要点）
1. 基礎（レベル1相当）値は `data_gen.textproto` の `base_hp` / `base_atk` / `base_def` に記載される（例: albedo の `base_hp: 1029.5856`）。
2. 任意レベル L における総合ベース値は次の式で算出される:
   - `total_base_hp(L) = base_hp * curveFactor(L, hp_curve)`
   - `curveFactor` は `pkg/model/curves.go` の `AvatarGrowCurveByLvl[L-1][HpCurveType]`（0-index）を使用。
3. ascensionで与えられる `promo_data` は「そのascensionレベルで加算される追加値（差分）」を表す。実務上はascension到達時点の総合値から前の段階の総合値を引いて差分を `promo_data.add_props.value` に格納している。これが asc テーブルの段差（例: 20→40 の +882.77655）と一致する。

### 生成パイプライン（このリポジトリの実行方法）
1. リポジトリの `Taskfile.yml` に `pipeline` タスクが定義されています。キャラクターデータ生成は以下のいずれかで実行します：
   - Taskfile を使用する場合（Task がインストールされている環境）:

```powershell
task pipeline
```

## AI自動生成フロー（機械実行仕様）

この節はAIが完全自動で `data_gen.textproto` を生成し、検算して埋め込み用ファイルを作るための機械実行可能な手順です。パイプラインソースが無くても、自動化スクリプトがこの手順に従えば同等の出力を再現できます。

### 入力（機械可読）
JSON または YAML。必須フィールドの例（JSON）:

```json
{
	"key": "albedo",
	"id": 10000038,
	"base_hp": 1029.5856,
	"base_atk": 19.551,
	"base_def": 68.2062,
	"hp_curve": "GROW_CURVE_HP_S5",
	"atk_curve": "GROW_CURVE_ATTACK_S5",
	"def_curve": "GROW_CURVE_HP_S5",
	"ascension_levels": [20,40,50,60,70,80,90],
	"elemental_bonus": {"50": 0.072, "80": 0.216}
}
```

### 処理アルゴリズム（実装手順）
1. JSON/YAML を読み込む。
2. `pkg/model/curves.go` の `AvatarGrowCurveByLvl` を参照して、各レベル L の成長係数を取得する（`AvatarGrowCurveByLvl[L-1][CurveType]`）。
3. `total_base_hp(L) = base_hp * curveFactor(L)` を計算（ATK/DEF も同様）。
4. 各 ascension レベルについて、`promo_data.value` を「その ascension の total - 直前 ascension の total（直前が無ければ base レベルの total）」で設定する。
5. `data_gen.textproto` をテンプレートに従って生成する（proto text 形式）。
6. `internal/characters/<key>/data_gen.textproto` に書き込む。
7. （オプション）テンプレート `internal/characters/<key>/<key>_gen.go` を自動生成して `//go:embed data_gen.textproto` を含める。

### 出力テンプレート（必須の抜粋）
```
id: <id>
key: "<key>"
stats: {
	base_hp: <base_hp>
	base_atk: <base_atk>
	base_def: <base_def>
	hp_curve: <hp_curve>
	atk_curve: <atk_curve>
	def_cruve: <def_curve>
	promo_data: {
		max_level: <asc_level>
		add_props: { prop_type: FIGHT_PROP_BASE_HP value: <hp_diff> }
		add_props: { prop_type: FIGHT_PROP_BASE_DEFENSE value: <def_diff> }
		add_props: { prop_type: FIGHT_PROP_BASE_ATTACK value: <atk_diff> }
		add_props: { prop_type: FIGHT_PROP_ROCK_ADD_HURT value: <elem_bonus> }
	}
}
```

### 検算と自動確認
生成後に以下ツールで検算します（リポジトリに追加済み）:

```powershell
go run ./tools/gen_char_stats --dir internal/characters/<key>
```

自動判定ロジック（スクリプト実装案）:
- `promo_data` の各 `max_level` について `computed_total(max_level) - computed_total(prev_level)` を計算し、`add_props.value` と差分の絶対値が閾値 (e.g. 1e-4) 以下なら PASS、そうでなければ FAIL を返す。

### CI への組み込み（推奨）
1. PR/merge pipeline に `go run ./tools/gen_char_stats --dir internal/characters/<changed_char>` を実行するステップを追加。
2. FAIL の場合は CI を失敗させレビューで修正を要求する。

### 補足
AI 実装はこの手順に従って JSON/YAML を生成し、上記コマンドを呼び出すだけで `data_gen.textproto` を作成・検算できます。

## 実行後に生成されるもの
- 各キャラクターのフォルダに `data_gen.textproto` と `*_gen.go`（`prototext.Unmarshal` を使って埋め込むファイル）が作成される。

## 検証手順（生成後に確認すべき点）
1. `data_gen.textproto` の `base_*` と `promo_data` の `value` を確認し、asc テーブルの値と一致するかチェックする。
   - 例: `20/20` → `40/40` の差が `promo_data` の `value`（`max_level:40` の `FIGHT_PROP_BASE_HP`）と一致すること。
2. 再計算での確認（ローカルで素早く検算する方法）:
   - `base_hp * AvatarGrowCurveByLvl[levelIndex][HpCurve]` を計算し、四捨五入して asc の値に一致することを確認する。
3. 生成後は `internal/characters/<char>/<char>_gen.go` の先頭に `// Code generated by "pipeline"` が付いていることを確認する。

## 推奨：CI に組み込むチェック
- `pipeline` 実行を CI に組み込み、生成後に `tools/check_keys` 的な簡易スクリプトで `data_gen.textproto` の `promo_data` が計算上の差分と一致することを検証することで、人為的なズレを防げます。

## 参考ファイル（実例）
- `internal/characters/albedo/data_gen.textproto` — 実際の base/promo がここに入っている。
- `internal/characters/albedo/albedo_gen.go` — 生成ファイル（`//go:embed data_gen.textproto` + `prototext.Unmarshal`）。
